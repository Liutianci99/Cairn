#  app 脚手架模板

template 仓库是个人玩具 app 的起点。新 app 从这里 clone，继承 `.claude/` 配置、`design/` 视觉规范与本文件的工作流约定。**所有玩具 app 开发必须遵循本文件。**

---

## 仓库内容

- `.claude/skills/` — `workflow`（接需求开场）、`prd`（PRD 撰写）、`commit`（提交推送）
- `.claude/hooks/pre-commit.sh` — 提交前静态检查门禁
- `design/` — 单文件 HTML 视觉规范，PRD 取色来源

## 工作流

玩具 app 开发有两个入口：从零开始的**新 app 立项**，已有项目的**接需求迭代**。两个入口都汇入同一套迭代循环与收尾。

### 新 app 立项

收到一个 app 想法，必须先评估「有没有必要做」，禁止直接开工。

- 加载 `brainstorming` 技能，确认需求与边界
- 调研同类产品，明确根本分歧；调研免费 API / 数据源，确认可行性
- 命名：英文名、人文气息、与产品语义相关（Stoic、Verba、Momento）
- 技术栈在 brainstorm 阶段确定，禁止预设，结论写入 PRD 与项目 `CLAUDE.md`
- 调用 `prd` 技能输出 `docs/prd.html`
- 按选定栈初始化脚手架

> 立项判断密集，以上是引导而非清单。每个 app 的取舍不同，按情况判断。

### 老项目接需求

开场四步见 `workflow` 技能。

### 迭代循环

每个需求走同一个小循环：

- **方案先行**：写代码前先给文字方案；多方案给选项对比表，标出推荐项与理由。UI 改动**必须**先出方案、等用户拍板，禁止直接落代码
- **实现与静态检查**：只实现需求范围内的改动，禁止顺手扩大范围；跑对应技术栈的静态检查（如 `flutter analyze`），必须 0 错
- **双重验收**：机器信号（静态检查与测试通过）+ 用户装机实测。装机验收由用户完成，Claude 不得替代确认

### 收尾

- 开功能分支 `<type>/<topic>`，改动提交到功能分支，提交调用 `commit` 技能，合入 main
- 构建命令依项目技术栈而定，记在各 app 自己的 `CLAUDE.md`

## 铁律

贯穿全流程，无论进行到哪一步都必须遵守。

- **破坏性操作必须授权**：删文件、写数据库、`git push --force` 等操作，必须先报告并取得用户明确同意，禁止自行执行
- **临时指令不外推**：用户针对当次的一次性指令（如「这批跳过 PR 直接合 main」）只对当次有效，禁止当作后续默认
- **memory 分作用域**：项目专属规范写入项目 `CLAUDE.md`，禁止写入全局 memory
- **commit message 用中文**：禁止英文 message

---

## 技术栈适配

`.claude/hooks/pre-commit.sh` 与 `commit` 技能默认按某一技术栈写死构建命令。新 app clone 后**必须**按实际技术栈改写构建与静态检查命令。

> 反例：Flutter 项目沿用写死 `mvn clean install` 的 hook → 提交门禁静默失效，改动未经检查即提交。
